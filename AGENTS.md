# Project Execution Policy

本项目后续对话和代码修改应遵循以下执行方针。

## 说中文

如题。

## 子代理的工作空间

如果你打算分派给子代理任务，你需要附带你的 session id，并为其分配 subagent id。

如果你是子代理，你不应创建 session id，也不应在 `.codex/conversations` 创建自己的工作空间，你应使用你的父代理的工作空间，并在其中的 `{subagent id}.md` 记录你的耐久结论，并在完成任务后向父代理汇报在你的工作中发现的值得记录的未来仍容易复发的项目级陷阱。

## 先审计，再修改

在修复缺陷或添加功能前，先阅读问题涉及的模块代码，以及可能存在相似逻辑的代码，并对当前实现做一次小范围审计：

- 当前结构是否清晰，职责是否集中在合理的模块内。
- 是否存在重复实现、过度耦合、隐式状态、全量轮询、无界增长、阻塞调用等未来容易触发的问题。
- 本次需求是否只是表层症状，背后是否有更健康的架构改造路径。
- 若改造风险较高，应先明确风险、收益和可分阶段方案，再动手。

## 主动处理架构问题

如果在阅读目标模块时发现明显架构问题，不应只完成最窄的功能补丁。应主动提出并优先处理低到中风险、收益明确的整理，例如：

- 模块化拆分或合并相似功能。
- 删除已经无实际价值的旧路径或重复 fallback。
- 将高频全量读取改成事件驱动、增量传输或有界缓存。
- 将杂乱的数据转换、状态拼装、展示逻辑移动到明确的边界。

## 优先保证架构清晰简洁

本项目仍处于早期阶段，在进行任何代码修改时，优先选择能降低长期复杂度的简化方案。除非用户明确提出发布兼容需求，不为尚未发布的旧路径、旧 API、旧数据格式或推测中的未来用例保留兼容层和扩展点。

## 一次聚焦一个模块

架构改进应保持递归发现问题的能力，但一次只修改一个相关模块或边界。若发现新的跨模块问题，应记录并向用户提出下一步建议，避免在同一轮中无界扩散。

## 审计文档与项目历史分工

- `docs/BACKEND_AUDIT.md` 与 `docs/FRONTEND_AUDIT.md` 是当前架构、模块边界、活跃问题和整改顺序的权威审计入口。完整审计时应重写过时结论，不累计已解决问题、旧代码行数、旧 PID 或旧测试数量。
- 审计文件前半部分应维护可供无上下文 session 快速定位模块的架构示意、关键数据流和持久化所有权；架构变化时同步更新对应审计文件。
- `.codex/project-history.md` 只记录近期聚焦的问题、近期发生的变化、仍有效的环境效果和下一步 handoff，不保存完整架构演进史。架构事实应写进审计或正式文档。
- `.codex/conversations/index.md` 只索引仍包含不可替代探索、原始数据或尚未写入权威文档的会话。会话是否可能继续不是保留理由；已完成且结论已提升到审计/history/lessons 的会话应归档。
- 整理持久状态时删除已经解决或失效的 active issue，合并重复 lesson；不要为了保留追溯性把实现时间线重新塞回 project history。
